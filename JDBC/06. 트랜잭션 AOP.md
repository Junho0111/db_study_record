# 트랜잭션 문제 해결 - 트랜잭션 AOP

### 프록시를 통한 문제 해결

트랜잭션 템플릿을 사용해도 서비스 계층에 비즈니스 로직과 트랜잭션 처리 기술 로직이 섞여있다는 문제가 남아있음. 이를 해결하기 위해 **스프링 AOP를 통한 프록시(Proxy)** 기술을 도입. 

* **프록시 도입 전**: 서비스가 직접 트랜잭션 시작과 종료 코드를 호출함. 

<br>

### 트랜잭션 프록시 코드 구조 예시 

* 실제로 이렇지는 않고 이런 구조로 돌아감 
<br>TransactionProxy 예시

```java
private MemberService target;
public void logic() {
    //트랜잭션 시작
    TransactionStatus status = transactionManager.getTransaction(..);
    try {
          //실제 대상 호출
          target.logic();
          transactionManager.commit(status); //성공시 커밋
        } catch (Exception e) {
            transactionManager.rollback(status); //실패시 롤백
            throw new IllegalStateException(e);
        }
}
```
> 위 로직이 @Transactional를 붙였을때 돌아가는 구조라고 보면되고 target로직이 애노테이션을 붙인 로직임

<br>

```java
public class Service {
    public void logic() {
    //트랜잭션 관련 코드 제거, 순수 비즈니스 로직만 남음
    bizLogic(fromId, toId, money);
    }
}
```

<br>
<br>

### 스프링이 제공하는 트랜잭션 AOP
* 스프링은 트랜잭션 AOP를 처리하기 위한 모든 기능을 제공하며,스프링 부트를 사용하면 트랜잭션 AOP를 처리하기 위해 필요한 스프링 빈들도 자동으로 등록해줌.

* 따라서 트랜잭션 처리가 필요한곳에 `@Transactional` 애노테이션만 붙여주면됨
<br>(스프링의 트랜잭션 AOP는 이 애노테이션을 인식해서 트랜잭션 프록시를 적용해줌)

<br>
<br>

> org.springframework.transaction.annotation.Transactional <== 스프링이 제공<br>
스프링 AOP를 적용하려면 Aspect, Advice, Potincut가 필요한데 스프링은 트랜잭션 AOP처리를 위해 아래 클래스를 제공하며 스프링부트 사용시 해당 빈들이 컨테이너에 자동 등록됨
<br>
<br>Aspect: BeanFactoryTransactionAttributeSourceAdvisor
<br>Potincut: TransactionAttributeSourcePointcut
<br>Advice: TransactionInterceptor

<br>
<br>

### 트랜잭션 AOP적용

```java
 @Transactional
 public void accountTransfer(String fromId, String toId, int money) throws SQLException {
    bizLogic(fromId, toId, money);
}
```
* @Transactional 애노테이션은 메서드에 레벨에 붙여도 되고, 클래스 레벨에 붙여도됨.
<br>(클래스에 붙이면 외부에서 호출 하능한 public 메서드가 AOP 적용 대상이됨)

<br>
<br>

### 스프링AOP 테스트 코드
* 다른 코드들과 달리 스프링 AOP를 적용한 메서드를 테스트 하려면 스프링 컨테이너가 필요함

* 예시
```java
 @Autowired
 MemberRepositoryV3 memberRepository;

 @Autowired
 MemberServiceV3_3 memberService;
```

<br>

```java
 @TestConfiguration
static class TestConfig {
    @Bean
    DataSource dataSource() {
        return new DriverManagerDataSource(URL, USERNAME, PASSWORD);
    }

    @Bean
    PlatformTransactionManager transactionManager() {
        return new DataSourceTransactionManager(dataSource());
    }

    @Bean
    MemberRepositoryV3 memberRepositoryV3() {
        return new MemberRepositoryV3(dataSource());
    }

    @Bean
    MemberServiceV3_3 memberServiceV3_3() {
        return new MemberServiceV3_3(memberRepositoryV3());
    }
}
```
* @SpringBootTest : 이 애노테이션이 있으면 테스트시 스프링 부트를 통해 스프링 컨테이너를 생성함. 그리고 테스트에서 @Autowired 등을 통해 스프링 컨테이너가 관리하는 빈들을 사용할 수 있음.

* @TestConfiguration : 테스트 안에서 내부 설정 클래스를 만들어서 사용하면서 이 에노테이션을 붙이면, 스프링 부트가 자동으로 만들어주는 빈들에 추가로 필요한 스프링 빈들을 등록하고 테스트를 수행할 수 있음.

* TestConfig : 
<br>1. DataSource 스프링에서 기본으로 사용할 데이터소스를 스프링 빈으로 등록함. 추가로 트랜잭션 매니저에서도 사용됨.
<br>2. DataSourceTransactionManager :  트랜잭션 매니저를 스프링 빈으로 등록한다
<br>(스프링이 제공하는 트랜잭션 AOP는 스프링 빈에 등록된 트랜잭션 매니저를 찾아서 사용하기 때문에 트랜잭션 매니저를 스프링 빈으로 등록해두어야 한다.)

# 선언적 vs 프로그래밍 방식

| 구분 | 선언적 트랜잭션 관리 | 프로그래밍 방식 트랜잭션 관리 |
| :--- | :--- | :--- |
| **방법** | `@Transactional` 애노테이션 사용 | `TransactionManager`나`TransactionTemplate` 직접 코딩 |
| **특징** | 비즈니스 로직과 기술 로직의 완전한 분리 | 기술 로직이 서비스 코드에 침범함 |
| **실무** | 대부분의 실무에서 사용 | 거의 사용되지 않음 (테스트 시 가끔 사용) |

<br>

# 데이터소스

* 기존에는 `DataSource`와 `TransactionManager`를 직접 스프링 빈으로 등록해야했음

```java
@Bean
DataSource dataSource() {
    return new DriverManagerDataSource(URL, USERNAME, PASSWORD);
}
@Bean
PlatformTransactionManager transactionManager() {
    return new DataSourceTransactionManager(dataSource());
}
```

<br>
<br>

* 위의 문제는 스프링 부트가 나오면서 많은 부분 자동화됨.
<br>(스프링 부트는 데이터소스를 스프링 빈에 자동으로 등록되며 이때 자동으로 등록되는 빈 이름은 dataSource)

<br>

# 스프링부트로 데이터소스 자동등록 하는 법
* 스프링 부트는 application.properties에 있는 속성을 사용해 데이터 소스를 생성하고 빈에 등록함.

```properties
spring.datasource.url=jdbc:h2:tcp://localhost/~/test
spring.datasource.username=sa
spring.datasource.password=
```
> 스프링 부트가 기본적으로 생성하는 데이터 소스는 커넥션풀을 제공하는 `HikariDataSource`임. 커넥션풀과 관련된 설정도 application.properties를 통해서 지정할 수 있음 
<br>기본적으로 spring.datasource.url에 값이 없으면 메모리DB를 생성함
t
* 따라서 데이터소스와 트랜잭션 매니저는 스프링 부트가 제공하는 자동 빈 등록 기능을 사용하는 것이 편리함

# 트랜잭션 매니저
* 스프링 부트는 적절한 트랜잭션 매니저(`PlatformTransactionManager`)를 자동으로 스프링 빈에 등록한다.
<br>자동으로 등록되는 스프링 빈 이름: `transactionManager`

* 스프링이 어떤 트랜잭션 매니저를 선택할지는 현재 등록된 라이브러리를 보고 판단함
<br>(예 JDBC를 기술을 사용하면 DataSourceTransactionManager
를 빈으로 등록)

<br>

JDBC와 JPA 둘다 사용하는 경우JpaTransactionManager를 등록함
<br>(JpaTransactionManager가 JDBC매니저 기능도 대부분 지원함)

<br>

# 트랜잭션 - DataSource와 transactionManager - 자동 등록

```java
    @TestConfiguration
    @RequiredArgsConstructor
    static class TestConfig {

        private final DataSource dataSource;

        @Bean
        MemberRepositoryV3 memberRepositoryV3() {
            return new MemberRepositoryV3(dataSource);
        }

        @Bean
        MemberServiceV3_3 memberServiceV3_3() {
            return new MemberServiceV3_3(memberRepositoryV3());
        }
    }
```

* 스프링 부트가 application.properties 에 지정된 속성을 참고해서 데이터소스와 트랜잭션 매니저를 자동으로 생성함.

* 생성자를 통해서 스프링 부트가 만들어준 데이터소스 빈을 주입 받을 수도 있음.

<br>

# 정리
* 스프링 AOP와 스프링 부트의 자동 설정으로  **순수한** 비즈니스 로직에만 집중하면서 안전하게 트랜잭션을 관리할 수 있음. 
* 결국 트랜잭션이 필요한 비즈니스 로직에는 @Transactional붙여서 사용해야함
