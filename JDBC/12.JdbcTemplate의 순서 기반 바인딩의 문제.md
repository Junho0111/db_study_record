# JdbcTemplate - 순서 기반 바인딩의 문제

## 개요
* `JdbcTemplate`의 단점 중 하나인 순서 기반 바인딩 문제
* 실무에서는 파라미터가 적지 않기에 실수 하는 경우가 있음
* 이를 해결하기 위해 파라미터 이름을 기준으로 매핑함

<br>

#### 기존 방식
```java
String sql = "update item set item_name=?, price=?, quantity=? where id=?";
template.update(sql, itemName, price, quantity, itemId);
```

* `?` 위치에 파라미터가 순서대로 바인딩됨
* 1번째 `?` → itemName
* 2번째 `?` → price
* 3번째 `?` → quantity
* 4번째 `?` → itemId

<br>

### 순서 기반의 위험성

만약 SQL 문의 순서를 변경하면? (price와 quantity 순서 변경)
```java
String sql = "update item set item_name=?, quantity=?, price=? where id=?";
template.update(sql, itemName, price, quantity, itemId);
```

* `quantity=price` (가격이 수량 칸에 들어감)
* `price=quantity` (수량이 가격 칸에 들어감)

<br>

#### 실무에서의 문제

| 상황 | 위험도 |
| --- | --- |
| 파라미터 10~20개 이상 | **극도로 높음** |
| 코드 리팩토링 중 SQL 순서 변경 | **높음** |
| 팀 협업 시 순서 실수 | **높음** |
| 버그 발견 후 데이터 복구 | **시간 소모 많음** |

> **핵심:** 코드 몇 줄 줄이는 편리함보다 **명확성과 안정성**이 훨씬 중요

<br>

## 문제 해결 - NamedParameterJdbcTemplate

* `?` 대신 **`:파라미터이름`** 을 SQL에 사용
* 파라미터를 **Map** 형태로 전달
* 순서가 아닌 **이름**으로 매칭되므로 안전함

#### 예시
```java
String sql = "update item set item_name=:itemName, price=:price, quantity=:quantity where id=:id";
```

**이점:**
* SQL 순서 변경해도 안전
* 어떤 파라미터가 어느 위치에 들어가는지 명확
* 유지보수 용이

<br>

### NamedParameterJdbcTemplate 이란?
* 스프링이 제공하는 JdbcTemplate의 확장 버전
* 이름 기반(:파라미터 명) 파라미터 바인딩을 지원
* 기본 JdbcTemplate의 모든 기능 + 이름 지정 기능

#### 사용 예시
```java
@Repository
public class JdbcTemplateItemRepositoryV2 implements ItemRepository {

    private final NamedParameterJdbcTemplate template;
    
    public JdbcTemplateItemRepositoryV2(DataSource dataSource) {
        this.template = new NamedParameterJdbcTemplate(dataSource);
    }
}
```

**주의:**
- 생성자에서 `DataSource`를 받아 `NamedParameterJdbcTemplate` 생성
- 스프링 관례상 이 방법을 가장 많이 사용

<br>


## 파라미터 전달 방식 3가지

#### 1. Map 방식

**가장 단순한 방식**

```java
@Override
public Optional<Item> findById(Long id) {
    String sql = "select id, item_name, price, quantity from item where id = :id";
    try {
        Map<String, Object> param = Map.of("id", id);
        Item item = template.queryForObject(sql, param, itemRowMapper());
        return Optional.of(item);
    } catch (EmptyResultDataAccessException e) {
        return Optional.empty();
    }
}
```

**특징:**
- 간단한 파라미터 1~2개일 때 적합
- `Map.of()`로 빠르게 작성 가능
- 명확성 높음

<br>

#### 2. MapSqlParameterSource 방식

**SQL에 최적화된 Map**

```java
@Override
public void update(Long itemId, ItemUpdateDto updateParam) {
    String sql = "update item " +
        "set item_name=:itemName, price=:price, quantity=:quantity " +
        "where id=:id";
    
    SqlParameterSource param = new MapSqlParameterSource()
        .addValue("itemName", updateParam.getItemName())
        .addValue("price", updateParam.getPrice())
        .addValue("quantity", updateParam.getQuantity())
        .addValue("id", itemId);
    
    template.update(sql, param);
}
```
>sql이 길어져 다음줄로 내려갈때 띄어씌기 주의해야함

**특징:**
- `SqlParameterSource` 인터페이스 구현
- 메서드 체인 지원 (깔끔한 코드)
- SQL 타입 지정 가능
- 파라미터 3~10개일 때 적합

<br>

#### 3. BeanPropertySqlParameterSource 방식

**자바빈 규약으로 자동 매핑**

```java
@Override
public Item save(Item item) {
    String sql = "insert into item (item_name, price, quantity) " +
                 "values (:itemName, :price, :quantity)";
    
    SqlParameterSource param = new BeanPropertySqlParameterSource(item);
    KeyHolder keyHolder = new GeneratedKeyHolder();
    template.update(sql, param, keyHolder);
    
    Long key = keyHolder.getKey().longValue();
    item.setId(key);
    return item;
}
```

**작동 원리:**
- 객체의 getter 메서드를 자동으로 인식
- `getItemName()` → `:itemName`
- `getPrice()` → `:price`
- `getQuantity()` → `:quantity`

**특징:**
- 가장 간편한 방식
- 파라미터 개수가 많을 때 효과적
- 단, 객체의 필드와 SQL 파라미터가 정확히 매칭되어야 함

<br>

#### 방식 선택 기준

| 상황 | 권장 방식 |
| --- | --- |
| 파라미터 1~2개, 간단한 쿼리 | **Map** |
| 파라미터 3~10개, 여러 객체에서 추출 | **MapSqlParameterSource** |
| 파라미터가 객체 필드와 일치 | **BeanPropertySqlParameterSource** |
| 일부 필드만 필요 | **MapSqlParameterSource** |

<br>

### RowMapper: BeanPropertyRowMapper

#### 기존 방식 - 수동 매핑
```java
private RowMapper<Item> itemRowMapper() {
    return (rs, rowNum) -> {
        Item item = new Item();
        item.setId(rs.getLong("id"));
        item.setItemName(rs.getString("item_name"));
        item.setPrice(rs.getInt("price"));
        item.setQuantity(rs.getInt("quantity"));
        return item;
    };
}
```

**문제점:**
- 매번 setter를 수동으로 작성
- 컬럼과 필드 이름 불일치 시 문제

<br>

#### 개선 방식 - BeanPropertyRowMapper 자동 매핑
```java
private RowMapper<Item> itemRowMapper() {
    return BeanPropertyRowMapper.newInstance(Item.class);
}
```

**작동 원리:**
- ResultSet의 컬럼명 자동 인식
- 자바빈 규약에 맞춘 setter 자동 호출
- 내부적으로 리플렉션 사용

<br>

## JdbcTemplateV2Config 설정

```java
@Configuration // ← 이걸 붙여줘야 새로운 버전의 리포를 주입 받을 수 있음
@RequiredArgsConstructor
public class JdbcTemplateV2Config {

    private final DataSource dataSource;
    
    @Bean
    public ItemService itemService() {
        return new ItemServiceV1(itemRepository());
    }
    
    @Bean
    public ItemRepository itemRepository() {
        return new JdbcTemplateItemRepositoryV2(dataSource);
    }
}
```

<br>

**동작:**
- `JdbcTemplateItemRepositoryV2` 빈 등록
- `ItemService`가 이를 주입받음
- 이제 모든 DB 작업에 이름 지정 파라미터 사용(JdbcTemplateItemRepositoryV2)

<br>

### 애플리케이션 설정 변경

## 추가 설명

#### snake_case ↔ camelCase 자동 변환

**자바 관례:** camelCase
```java
itemName, maxPrice, memberName
```

**DB 관례:** snake_case
```sql
item_name, max_price, member_name
```

**BeanPropertyRowMapper의 마법:**
```sql
select item_name, price from item
-- item_name → setItemName() 자동 호출
-- price → setPrice() 자동 호출
```

> **중요:** 언더스코어(`_`)는 자동으로 카멜케이스로 변환됨!

<br>

#### 컬럼과 필드 이름 완전히 다른 경우 - 별칭 사용

```sql
-- 데이터베이스
select member_name from members

-- 객체
private String username;  // itemName과 완전히 다름

-- 해결책: 별칭 사용
select member_name as username from members
```

**예시:**
```java
String sql = "select member_name from members"; // 작동 안 함

String sql = "select member_name as username from members"; // 작동 함
```
