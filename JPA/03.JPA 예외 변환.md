# JPA 예외 변환

<br>

## 예외 변환 메커니즘

**JPA 원본 예외:**
* `PersistenceException` 및 하위 예외
* `IllegalStateException`, `IllegalArgumentException`

**변환 후:**
* Spring의 `DataAccessException` (추상화)

<br>

## @Repository의 기능

```java
@Repository
@Transactional
public class JpaItemRepositoryV1 implements ItemRepository {
    // ...
}
```

**@Repository 역할:**
1. 컴포넌트 스캔 대상
2. **예외 변환 AOP의 대상**

<br>

## 동작 흐름

```
서비스 계층
    ↓
JpaItemRepositoryV1 (@Repository)
    ↓
    ↓ (예외 변환 AOP 프록시 적용)
    ↓
EntityManager
    ↓
    ↓ (JPA 예외 발생)
    ↓
PersistenceException
    ↓
    ↓ (AOP가 자동 변환)
    ↓
DataAccessException
    ↓
서비스 계층에서 처리(@Service)
```

**변환 담당:**
* `PersistenceExceptionTranslator` : JPA 예외 → Spring 예외
* `PersistenceExceptionTranslationPostProcessor` : AOP 프록시 등록

<br>

### 변환 전

```
EntityManager
    ↓
PersistenceException (JPA 예외)
    ↓
서비스 계층 (처리 불가)
```

### 변환 후

```
EntityManager
    ↓
PersistenceException (JPA 예외)
    ↓
    ↓ (AOP 프록시가 자동 변환)
    ↓
DataAccessException (Spring 예외)
    ↓
서비스 계층 
```

>설명을 위해 @Service에서 예외를 처리한다고 적었지만 예외는 <br>
사용자 정의 예외 클래스 또는 @ExceptionHandler, @RestControllerAdvice,@ControllerAdvice를 사용하여 처리하는것을 권장 함
